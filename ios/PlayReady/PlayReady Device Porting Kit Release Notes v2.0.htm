Release Notes
Microsoft® PlayReady® Device Porting Kit v2.0

September 2011

The Microsoft® PlayReady® Device Porting Kit (Device PK) v2.0:

·         Enables playback of ASF encrypted content protected by the WMDRM10 technology using PlayReady based XMR licenses.

·         Provides support for the Silverlight™ browser plug-in; specifically, this version of the porting kit will enable playback of ASF encrypted content tied to a Silverlight DRM license.

Hardware and software requirements

PlayReady Device Porting Kit (Device PK) is designed to work on various platforms and compilers.

Hardware

There are no specific hardware requirements for the Device PK development environments.

Target devices must have an original equipment manufacturer layer that is compliant and robust. Target devices must also support certificate revocation.

Compilers

Device PK works on compilers that support ANSI C. Compiler definitions that are included out-of-the-box include Microsoft C and Gnu C.

What's New

This section describes the most significant differences between PlayReady Device Porting Kit 1.2 and 2.0.

Removed Features and Functionality

The following features and functionality have been removed in PlayReady Device Porting Kit 2.0 (Device PK 2.0). They are no longer supported.

•   MSDB – The Microsoft Device Bridge has been removed.

•   Build Profiles – All build profiles except the OEM profile and test profiles have been removed. Other profiles that previously shipped with Device PK were designed for testing only, and they were removed to eliminate confusion surrounding upon which profile an OEM should base their implementation.

•   16 Bit Addressing – Platforms that do not support 8-bit addressing are no longer supported.

Revocation Support is Required

Building a device without support for revocation using Device PK is no longer possible. All devices must support revocation.

Key Files are Required

All devices built on Device PK 2.0 must use Key Files for key management. Refer to Key Files and Key History in the Programming Guide for more information about key management.

Core Device Assets Have Changed

In previous versions of Device PK, the core device assets needing to be protected by the OEM were the device private keys. These needed to be individually protected by the OEM.

In Device PK 2.0, the device private keys are stored in the Key File. These keys are themselves protected by AES Keys. Thus, the core device assets needing to be protected by the OEM are now these AES keys. This is a paradigm shift in what the OEM must protect in order to meet the PlayReady Compliance and Robustness Rules. Refer to Key Files and Key History in the Programming Guide for more information about these AES keys.

Feature Selection at Link-Time

In Device PK 2.0, most features are no longer enabled or disabled by changing compile-time flags (#defines). Instead, they are selected by compiling linking to different versions of the same library – “stub” to disable a feature versus “real” to enable a feature. This results in significantly smaller code size and better modularization.

Better Support for Hardware Crypto Offload

All cryptographic functionality, including multi-block AES, has been moved into the OEM layer. This enables OEMs to more easily replace it with hardware implementations where available.

Better Support for Big Endian Platforms

Device PK 2.0 has been more thoroughly tested on platforms with Big Endian architectures. Such platforms are now fully supported.

Support for PlayReady-Only Devices

All WMDRM functionality is now enclosed within a feature that can be selected at link-time. If the WMDRM feature is disabled by linking to the “stub” module, then the device will only support PlayReady features and will not require WMDRM certificates.

Hashed Data Store (HDS) Incompatibility

A device that had a previous version of the Device PK installed cannot be upgraded to Device PK 2.0 without deleting all persisted data on the device. Device PK 2.0 is incapable of consuming previous versions of the Hashed Data Store (HDS) where persisted data, including licenses for content, is stored.

Support for Live TV

Functionality designed specifically to support devices that consume live IP TV has been added to Device PK 2.0. This includes the following:

•   Scalable Licenses – Licenses designed to support the simultaneous broadcast of large numbers live IP TV channels are supported by Device PK 2.0. License Acquisition servers that use the PlayReady Server SDK 2.0 can create these licenses. Such licenses have ample performance to support live IP TV scenarios and provide adequate scalability on the license server.

•   Key Rotation – Content that changes cryptographic keys during playback is now supported by Device PK 2.0. The packaging tools included with PlayReady Server SDK 2.0 can create such content. Such content prevents a malicious user from cancelling their subscription and continuing to access live IP TV content and enables scenarios, such as sports blackout.

•   Version 4.1 Content Headers – Content headers designed to accompany Key Rotation content are now supported by Device PK 2.0. The packaging tools included with PlayReady Server SDK 2.0 can create such headers. Such headers enable the device to distinguish between content with a fixed key and content that rotates keys during playback.

 

Known issues / Limitations

Installation

Installing v2.0 of the PlayReady Device Porting Kit will not uninstall any v1.2 pre-release or final releases of the PlayReady Device Porting Kit. To have a single version of the Device PK installed, use Add/Remove Programs to manually uninstall any previous versions. Versions 1.2 and 2.0 can both be installed at the same time.

Running tests on Windows 7 or Windows Vista

The porting kit tests should be run in a command prompt window opened in elevated mode when running on Windows 7 or Windows Vista.

Initiators

The code under the directory Test\tools\initiators\sampleapp is prone to known issues and is not recommended for commercial usage.

ASF parsing

PlayReady applications performing ASF parsing should look for the PlayReady object in the ASF top-level header object as well as in the ASF header extension object.

Code Profiling

The default ANSI implementation of the profiling (DRM_SUPPORT_PROFILING) feature does not compile on systems without native 64-bit types. To resolve this issue, replace DRM_UI64LITERAL with DRM_UI64HL in all locations in source\oem\ansi\oemprofile.c.

Change:    return DRM_UI64LITERAL( 0, clock() );

To:    return DRM_UI64HL( 0, clock() );

Change:    return DRM_UI64LITERAL( 0, CLOCKS_PER_SEC );

To:    return DRM_UI64HL( 0, CLOCKS_PER_SEC );

Tracing

The default ANSI implementation of the tracing (DRM_SUPPORT_TRACING) feature does not produce valid trace files on Big-Endian ports. To resolve this issue, convert all DRM_WORDs, DRM_DWORDs, and enumerations being written to the trace file from Big-Endian to Little-Endian before they are written to the file in source\oem\ansi\trace\oemtrace.c.
